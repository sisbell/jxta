Participating in the JXSE (JXSE) Project

(Updated 2007-12-7 bondolo@dev.java.net)

Introduction

We encourage everybody to participate in the project. There are a number of ways community members can contribute to JXSE. Each level of contribution requires additional commitment of your time. Most people usually start out with smallercommitments and then work their way up as their personal schedules and priorities permit.

  • Tester : Found a problem in a previous release? Interested in a feature in an upcoming release? There is no better way to make sure it's perfect than to try it out for yourself! You can also contribute by creating new tests, adding to the benchmarking suite and to the JDF distributed testing project.

    In order to be a tester you should subscribe to the JXTA dev discussion forum. See JXSE Discussion Forums. You also want to become a project Observer or Developer in order to be able to modify the Issuezila database. You may also wish to become a Developer on the JXTA Benchmarking project in order to be able to add additional tests. See the JXTA Governance and Request Project Role pages to sign up!

  • Reviewer : A JXTA expert but don't have a lot of time to work on tasks? Trying to learn more about how JXTA is put together? Become a reviewer! Reviewers commit to devoting time to monitor and provide feedback on specific tasks. The most careful review is usually needed before code is committed for a task, but you can provide insight or comment on the ongoing progress of a task at any time. Scan the Issuezilla database for tasks which interest you or look for issues with the keyword "Review". Sign your self up by adding your name to the issue "cc:" list!

    In order to become a reviewer you should be an Observer or Developer on the JXSE project. See the JXTA Governance and Request Project Role pages to sign up!

  • Task Leader : Got a great idea? You can lead tasks for inclusion within a JXSE release. This usually involves coding, preparing documentation and unit tests, submitting code for pre-commit review and working with testers and community members to integrate your task into the release. You will also normally be expected to be willing to act as a reviewer for other tasks.

    In order to become a reviewer you should be an Developer on the JXSE project. See the JXTA Governance and Request Project Role pages to sign up!

  • Release Team : You can join the team which manages the release cycle for JXSE. This usually involves a lot of regression testing, integration testing, code commit reviews, writing release notes, tracking tasks and bugs. Early in the release cycle you will work with task leaders to establish the tasks to be included in the upcoming release. Most release team members will be task leaders for one or more tasks that will be part of the upcoming release.

    In order to become a member of the release team you should be Developer or Project Owner on the JXE project. See the JXTA Governance and Request Project Role pages to sign up!

Tasks

Each JXSE release will try to advance the project by adding new features or improving upon existing features. Each improvement is tracked as a task so that it's progress can be monitored and it can be discussed more effectively by the community.

Each task is associated with the particular release, the release in which it is expected to be included. Each task also has a leader. The leader of a task is reponsible for reporting information about the progress of tasks either to the appropriate JXTA mailing lists or via Issuezilla. The task leader is also likely the person completing the work.

Any community member can create a new task for a release. For now, creating new tasks requires that you also commit to leading the task and completing the work. You can also volunteer to work on existing tasks or create tasks based upon suggestions by others.

New tasks should be proposed on the JXTA Dev mailing list. Discussion about existing tasks should be discussed on its Issuezilla issue or on the JXSE DEV Forum.

Committer Policies

The JXSE project has evolved a flexible, informal policy for how code is developed and committed. Most JXTA projects use the Apache Voting Rules process with either a "Review Then Commit" (RTC) or "Commit Then Review" (CTR) policy. These "rules" apply to the Platform, CMS and, with exceptions noted below, to the Shell. All of the JXSE projects use the CTR process during normal development but between beta and final releases we require that any significant issues be reviewed before commit.

All code to be committed to the JXTA-JXSE, CMS and Shell projects must be compatible with Java SE 5.0 and with the JXSE Coding Conventions. You are encouraged to test your submissions on a variety of computer platforms and with a variety of Java Virtual Machines.

These conventions have always been applied informally. Good judgement and selective application has always been preferred to strict adherence.

Creation of Bugzilla Issues

Historically, the Platform project has made extensive use of the Issuezilla database (Platform Issue Tracker) as a focus for development. "Bug" issues are commonly created for defects, new features, patches and tasks. Issues provide a convenient forum for discussion of defect reports, design proposals, patches and resolutions.

There are some types of issues and bugs which traditionally have not resulted in a formal bugzilla issue being created. These normally include; updates to JavaDoc, comments, spelling corrections, logging changes, most PMD nits, web site changes, most changes to build scripts and packaging.

For a little larger changes, such as minor code cleanups, some more complex PMD nits, and very narrow minor bug fixes (e.g. throw IAE at the beginning of a method rather than allowing an NPE 10 lines later) we have generally done limited peer reviews and not created a Bugzilla issue. The peer review is normally done informally via email, IM, IRC, phone, over the shoulder, etc. These peer reviews are not intended to be a thorough indepth reviews, rather they are "No, you're not crazy" or "Maybe you should create an issue for this" checks.

For any behaviour changes, API signature changes, most bug fixes, anything potentially warranting discussion a Bugzilla issue must be created. The discussion and review process varies greatly depending upon the nature and complexity of the isssue involved. Developers are responsible for seeking appropriate review and comment before commit as appropriate. When in doubt, seek more review and comment.

Important Exceptions and the Shell Project

There are some exceptions to the preceeding rules. In particular, the Shell project has never made much use of the Issuzilla database. Committers to the Shell project are encouraged to simply follow their best judgement and be mindful not to break features. Shell committers should also pay attention to release plans and avoid committing big changes immediately before a planned release.

Any test directories are also special in that unlike all other portions of a project, you are actively encouraged to commit code which fails against the current source tree. All commits to the test directory should indicate a Bugzilla issue which should describe the nature of the test and the cause of the breakage.
